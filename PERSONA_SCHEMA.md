# Persona Schema

HumorRL 的长内容主线，下一阶段不再从“主题”直接开始，而是从**人物**开始。

核心原则：

- 先有一个接近真实的人
- 再围绕这个人去讲主题
- Judge 和改写都要检查“这像不像这个人会说的话”

## 为什么需要 Persona Schema

如果人设只有一句标签，比如：

- “呼兰风格”
- “文本型演员”
- “冷静观察”

那生成器很容易写出一篇“像某种风格”的稿子，但不像**一个具体的人**。

而真实可持续的长内容，应该至少满足：

1. 有清晰社会位置
2. 有稳定的人生处境
3. 有持续的情绪底色
4. 有固定的幽默发动方式
5. 有一致的观众关系

## Schema 字段

### 1. `one_line`

一句话定位。  
要求：一句话就能说清这个人是谁、靠什么好笑。

### 2. `social_identity`

他的社会身份、阶层位置、典型环境。  
要求：不能抽象，要让人能想象他平时活在哪里。

### 3. `life_context`

他长期处于什么生活处境。  
要求：这是内容源头，不是履历表。

### 4. `life_experience`

几条具体人生经验。  
要求：这些经验应当能稳定产出内容。

### 5. `core_drive`

他为什么要讲这些东西。  
要求：不是“为了搞笑”，而是这个人内在为什么总在看这些事。

### 6. `emotion_base`

长期情绪底色。  
要求：写“克制 / 疲惫 / 清醒 / 热烈 / 委屈 / 锋利”这类长期状态，不写瞬时情绪。

### 7. `voice`

说话方式。  
要求：像一个真人会说的话，不是修辞教科书。

### 8. `audience_relationship`

他和观众是什么关系。  
常见类型：

- 一起看荒诞
- 服务型取悦
- 挑衅型对抗
- 认同型召集

### 9. `humor_engine`

这个人的笑是怎么发动的。  
示例：

- 观察型共鸣
- 强画面行为
- 认知戳破
- 轻微反讽
- 情绪反弹

### 10. `preferred_topics`

这类人更容易稳定产出内容的主题。

### 11. `avoid`

这类人不该写成什么样。  
这是防止人设漂移最重要的字段之一。

### 12. `structure_preferences`

这个人更适合怎样的结构。  
例如：

- 开场快
- 中段案例递进
- 结尾轻拧
- 不适合长铺垫

### 13. `ending_preferences`

这个人适合怎样收尾。  
结尾方式应该和人格一致。

### 14. `persona_consistency_checks`

Judge / Strategist / Rewrite 以后都可以用的检查问题。  
要求：写成可判断的问题，而不是口号。

## 实施建议

### 生成阶段

先给生成器完整 persona，再给主题。  
生成器要优先回答：

- 这个人为什么会讲这个主题
- 他会从什么角度切进去
- 他会怎么收尾

### Judge 阶段

Judge 未来不只看：

- 好不好笑
- 结构成不成立

还要看：

- 像不像这个人会说的话
- 有没有脱离这个人的世界

### Rewrite 阶段

改 weakest segment 时，不能只补笑点，还要补：

- 这个人会不会这么说
- 这段和整个人设是否一致

## 当前落地状态

当前项目已经开始支持把结构化 persona profile 注入长稿生成器。  
也就是说，后续长稿主线会逐步从：

`topic -> generate set`

升级成：

`persona -> topic -> persona-driven set`
